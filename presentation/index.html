Lean in practice

  • Functional programming language
  • Interactive theorem prover
  • Used for mathematics, software verification, and teaching
  • Best understood as a PL story, not a product tour

Part II

Lean starts from familiar ground

Before dependent types and proofs, Lean already looks like a typed functional language.

Static vs Dynamic

First contact with Lean

`#eval`

Ask Lean to compute the expression.

Static view

What kind of thing is this expression?

Dynamic view

What does this expression compute to?

Explicit contract

(1 + 2 : Nat) and (1 - 2 : Int) make the intended type visible.

Definitions

Start with a named value


def hello : String := "Hello"

`def` introduces a name

`:=` means the name is defined by the expression on the right.

Why this matters

We are moving from raw expressions to a typed environment of named meanings.

Definitions

Then turn the definition into a function


def add1 (n : Nat) : Nat := n + 1

Input

(n : Nat) introduces an argument named n of type Nat.

Output

The return type is part of the function's contract, not decoration.

Definitions

Ordinary typed programming still scales up


def addTwo (n : Nat) (k : Nat) : Nat := n + k

Multiple parameters

Nothing mysterious yet. This is familiar typed functional programming.

Key baseline

Types already describe a useful input/output contract before we add richer invariants.

Inductive Data

Data is defined by its constructors


inductive Bool where
  | false : Bool
  | true : Bool

Formation rule

An inductive type tells us the legal shapes of its values.

`Bool` is intentionally simple

Two constructors, no recursion, just enough structure to explain the idea cleanly.

Pattern Matching

Constructors build data. `match` consumes it.


def notBool (b : Bool) : Bool :=
  match b with
  | Bool.false => Bool.true
  | Bool.true => Bool.false

Elimination rule

Pattern matching chooses a result according to the constructor that built the value.

Programming pattern

The structure of the program mirrors the structure of the data.

Recursive Data

Natural numbers are inductive too


inductive Nat where
  | zero : Nat
  | succ (n : Nat) : Nat

Self-similar structure

Nat is built from zero and repeated applications of succ.

Consequence

Recursive data leads naturally to recursive programs.

Structural Recursion

Recursive programs follow the same shape


def double (n : Nat) : Nat :=
  match n with
  | Nat.zero => Nat.zero
  | Nat.succ k => Nat.succ (Nat.succ (double k))

Base case

Handle Nat.zero.

Step case

Recur on the structurally smaller value k.

Checkpoint

What ordinary types already give us

Expressions

#check and #eval separate static and dynamic views.

Programs

def, function types, pattern matching, structural recursion.

Data

Inductive types define legal shapes, and programs consume those shapes.

Bridge to the next step

Part III

This is where Lean stops feeling ordinary

Ordinary types classify values. Dependent types can preserve relationships between them.

Programs Types Proofs

Motivation

Ordinary types are sometimes too weak

`zip` with ordinary lists


List α → List β → List (α × β)

The element types are remembered. The relationship between the two lengths is not.

If the lengths differ, what should happen?

  • Ignore extra data?
  • Raise an error?
  • Return an option?
  • Or make mismatched inputs unrepresentable?

Dependent types are about keeping more of that relation visible to the type checker.

Dependent Function

A return type can depend on the input value


(b : Bool) → if b then Nat else String

The shock point

The familiar if ... then ... else ... has moved inside the type.

Meaning

The result type itself changes with the input value.

Dependent Function

Case analysis now refines types too


def natOrStringThree (b : Bool) : if b then Nat else String :=
  match b with
  | true => (3 : Nat)
  | false => "three"

Computation + typing

Pattern matching is no longer only choosing a branch at runtime. It also sharpens the static type in each branch.

New flow of information

Values can refine types.

Indexed Data

What if a collection remembered its length?


inductive Vect (α : Type) : Nat → Type where

Not one type

Vect α n is a family of types indexed by a natural number.

Interpretation

Read it as "vectors of elements of type α and length n."

Indexed Data

The constructors carry the length story


inductive Vect (α : Type) : Nat → Type where
  | nil : Vect α 0

Empty vector

`nil` can only construct a vector whose length index is exactly 0.

Invariant in the type

The constructor itself tells Lean what length has been built.

Indexed Data

Adding one element changes the index too


inductive Vect (α : Type) : Nat → Type where
  | nil : Vect α 0
  | cons : α → {n : Nat} → Vect α n → Vect α (n + 1)

Constructor discipline

Adding an element changes the type from length n to length n + 1.

Syntax note

{n : Nat} is an implicit argument, kept off most later uses so the slides stay readable.

Primary Wow Moment

`Vect.zip` has no mismatched-length cases


def Vect.zip : Vect α n → Vect β n → Vect (α × β) n
  | Vect.nil, Vect.nil => Vect.nil
  | Vect.cons x xs, Vect.cons y ys =>
      Vect.cons (x, y) (Vect.zip xs ys)

Ask the room

Where are the unequal-length branches?

Answer

They are not missing by accident. The type already ruled them out.

Payoff

A runtime check has been moved into a compile-time guarantee.

Part IV

Theorem proving is the same story pushed further

A theorem statement is another typed object inside the same language. A proof is evidence for it.

Programs Types Proofs

Propositions

`Bool` and `Prop` are not the same thing

Boolean values


#check true
#check false

These are data values of type Bool.

Propositions


#check True
#check False

These live in Prop, where theorem statements are expressed.

Logic is not entering from outside the language. It is represented inside the same type-theoretic framework.

A Theorem Statement

Lean asks for evidence, not persuasion


theorem keepFirst (p q : Prop) (hp : p) (hq : q) : p := ?

Read it like a strict signature

Assume propositions p and q, plus evidence hp : p and hq : q. Produce evidence for p.

The hole matters

It makes Lean's demand explicit: fill in a term of the required type.

A Theorem Statement

The smallest proof looks like returning a value


theorem keepFirst (p q : Prop) (hp : p) (hq : q) : p := hp

Proof as program

The goal asked for evidence of type p, and hp already has that type.

Why this lands

It does not feel like a separate proof language. It feels like returning the needed value.

Structured Evidence

Conjunction behaves like data with constructors and projections


#check And.intro
#check And.left
#check And.right

`And.intro`

Build evidence for p ∧ q from evidence for each side.

`And.left` / `And.right`

Project the left or right piece back out.

Same pattern again

Build, extract, reassemble. The proof story mirrors typed programming.

Proof Terms

A larger proof is still just composition


theorem and_commutative (p q : Prop) (hpq : p ∧ q) : q ∧ p :=
  And.intro (And.right hpq) (And.left hpq)

Read it like a recipe

Take the right part, take the left part, then rebuild the conjunction in the opposite order.

Conceptual point

Logical connectives behave like structured data with introduction and elimination rules.

Tactic Mode

The same proof can be built interactively


theorem and_commutative_tactic (p q : Prop) (hpq : p ∧ q) : q ∧ p := by
  apply And.intro
  exact And.right hpq
  exact And.left hpq

`by` opens tactic mode

Lean can guide proof construction step by step.

What did not change

The theorem is the same. The underlying proof object is the same kind of thing.

Structural Unity

Recursion and induction have the same shape

Recursive program on `Nat`


def f : Nat → ...
  | Nat.zero => ...
  | Nat.succ k => ... f k ...

Induction proof on `Nat`


theorem h : ∀ n : Nat, P n
  | Nat.zero => ...
  | Nat.succ k => ... h k ...

That parallel is not a coincidence. It comes from the same inductive structure we started with in the programming part of the talk.

Part V / Synthesis

The core idea of Lean

01

Start with ordinary typed programming

Expressions, functions, data, and recursion are already enough to build the baseline model.

02

Let types track more meaning

Dependent types weaken the boundary between values and types so invariants survive at compile time.

03

Read proofs as programs

A specification becomes a type, and a proof is a term inhabiting that type.

Lean is not a bag of advanced features. It is the unification of programming, typing, and proving in one language.
